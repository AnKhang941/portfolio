Java OOP: Chuyện cái khuôn bánh và chiếc bánh quy (Class vs Object) - Hiểu một lần nhớ cả đời

December 21, 2025 By Nguyễn An Khang

Chào bạn, đây là phiên bản "full không che" cho chủ đề Class và Object. Đây là bài viết nền tảng cực kỳ quan trọng, vì nếu mất gốc chỗ này thì học lên các phần sau (Kế thừa, Đa hình) sẽ rất khổ sở.

Bài viết này đi sâu vào cả tư duy thiết kế lẫn cách máy tính xử lý trong bộ nhớ, giúp người đọc không chỉ biết code mà còn hiểu máy tính đang làm gì.

Hồi mới chân ướt chân ráo học Java, mình cứ lơ mơ mãi về hai khái niệm: Class (Lớp) và Object (Đối tượng). Sách giáo khoa thì định nghĩa: "Class là bản thiết kế, Object là thể hiện cụ thể...". Nghe rất trừu tượng và khó nuốt.

Mãi sau này đi làm, mình mới ngộ ra một chân lý đơn giản. Đừng nghĩ phức tạp, hãy nghĩ về chuyện làm bánh. Hôm nay, mình sẽ giải thích lại cho các bạn theo phong cách "Bếp trưởng" nhé.

1. Class (Lớp) - Chính là cái khuôn bánh

Hãy tưởng tượng bạn muốn mở tiệm bán bánh quy hình ngôi sao. Bạn không thể nặn tay từng cái một được, vì cái thì méo, cái thì tròn, lại tốn thời gian. Bạn cần một cái Khuôn (Cutter).

Trong lập trình, Class chính là cái khuôn đó.

Đặc điểm: Cái khuôn quy định hình dáng (ngôi sao), kích thước, thành phần.

Quan trọng: Bản thân cái khuôn không ăn được. Nó chỉ là một khái niệm, một bản mô tả về cái bánh thôi.

Cấu tạo của một Class: Một cái khuôn (Class) thường có 2 phần chính:

  • Thuộc tính (Attributes/Fields): Bánh màu gì? Vị gì? Nặng bao nhiêu? (Dữ liệu).
  • Phương thức (Methods/Behaviors): Bánh này dùng để làm gì? (Hành động).
// Đây là cái khuôn (Class)
public class BanhQuy {
    // 1. Thuộc tính (Đặc điểm)
    String mauSac;
    String huongVi;
    double giaTien;

    // 2. Phương thức (Hành động)
    void gioiThieu() {
        System.out.println("Tôi là bánh vị " + huongVi + ", màu " + mauSac);
    }
    
    void an() {
        System.out.println("Rộp rộp... Ngon quá!");
    }
}

2. Object (Đối tượng) - Chính là chiếc bánh thật

Bây giờ, bạn lấy bột, đè cái khuôn BanhQuy lên bột. Bùm! Bạn có một chiếc bánh thật. Trong lập trình, hành động "dập khuôn" đó gọi là Khởi tạo (Instantiation) với từ khóa thần thánh new.

Chiếc bánh vừa tạo ra chính là Object.

Đặc điểm: Nó là vật thể thật, chiếm chỗ trong không gian (bộ nhớ RAM).

Quan trọng: Từ 1 cái khuôn, bạn có thể tạo ra vô số cái bánh.

public class NhaBep {
    public static void main(String[] args) {
        // Dập khuôn lần 1: Tạo ra chiếc bánh cho khách A
        // 'new BanhQuy()' là lúc cái bánh thực sự được tạo ra
        BanhQuy banhCuaA = new BanhQuy(); 
        banhCuaA.mauSac = "Vàng";
        banhCuaA.huongVi = "Bơ sữa";

        // Dập khuôn lần 2: Tạo ra chiếc bánh cho khách B
        BanhQuy banhCuaB = new BanhQuy();
        banhCuaB.mauSac = "Nâu";
        banhCuaB.huongVi = "Socola";
        
        // Gọi hành động
        banhCuaA.gioiThieu(); // In ra: Bánh vị Bơ sữa
        banhCuaB.gioiThieu(); // In ra: Bánh vị Socola
    }
}

Sự khác biệt cốt lõi:

Nếu bạn bẻ đôi cái bánh của khách A (banhCuaA), cái bánh của khách B (banhCuaB) có bị vỡ theo không? Đương nhiên là Không.

Object độc lập hoàn toàn với nhau về dữ liệu, dù chúng được sinh ra từ cùng một Class.

3. Chuyện gì xảy ra trong bộ nhớ? (Deep Dive)

Đây là phần phân loại Dev "thợ" và Dev "kỹ sư". Hiểu được bộ nhớ, bạn sẽ tránh được lỗi NullPointerException huyền thoại.

Khi bạn viết dòng code: BanhQuy banh = new BanhQuy();

Máy tính thực hiện 2 việc ở 2 vùng nhớ khác nhau:

  • Heap (Vùng nhớ Đống): Máy tính dành một khoảng trống lớn để lưu trữ toàn bộ thông tin của chiếc bánh thật (new BanhQuy()). Đây là Object.
  • Stack (Vùng nhớ Ngăn xếp): Máy tính tạo ra một biến nhỏ tên là banh. Biến này không chứa cái bánh, nó chỉ chứa cái địa chỉ (hoặc cái điều khiển từ xa) trỏ đến cái bánh nằm bên Heap. Đây là Reference (Tham chiếu).

Ví dụ dễ hiểu:

  • Object (Heap): Là cái Tivi xịn sò đặt trong phòng khách.
  • Variable (Stack): Là cái Remote cầm trên tay.
  • Dấu bằng (=): Là hành động kết nối Remote với Tivi.

Hệ quả thú vị:

BanhQuy banh1 = new BanhQuy(); // Tivi 1, Remote 1
BanhQuy banh2 = banh1;         // Copy Remote 1 đưa cho Remote 2

banh2.mauSac = "Đỏ"; 
// Hỏi: banh1.mauSac bây giờ là màu gì?
// Đáp: Màu Đỏ!

Tại sao? Vì banh1 và banh2 là hai cái Remote khác nhau, nhưng cùng trỏ vào MỘT cái Tivi duy nhất. Ai bấm chuyển kênh thì người kia cũng thấy kênh đổi.

4. Tại sao cần OOP? Code kiểu cũ không được à?

Bạn có thể hỏi: "Sao phải rắc rối tạo Class làm gì? Tôi khai báo biến rời rạc cũng được mà?"

Được, nhưng hãy tưởng tượng bạn quản lý 1.000 cái bánh.

Cách cũ (Procedural): Bạn phải quản lý 1.000 biến màu, 1.000 biến vị, 1.000 biến giá tiền. String mauBanh1, mauBanh2... mauBanh1000; -> Điên đầu ngay!

Cách OOP: Bạn chỉ cần quản lý một danh sách List<BanhQuy>. Mọi thông tin màu sắc, hương vị đã được đóng gói gọn gàng trong từng object rồi. Code sạch, dễ bảo trì, dễ nâng cấp.

Kết luận

Tóm lại, để giỏi Java OOP, hãy nhớ kỹ câu thần chú này:

  • Class: Là Khái niệm, là Khuôn mẫu (File .java bạn viết).
  • Object: Là Thực thể, là Dữ liệu thật đang chạy trong RAM (new ClassName()).

Lần tới khi bạn new một cái gì đó, hãy nhớ rằng bạn đang dập một cái khuôn để tạo ra một chiếc bánh mới trong bộ nhớ Heap nhé!
