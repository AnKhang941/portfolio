Nghệ thuật viết Commit Message: Đừng để đồng nghiệp muốn "tác động vật lý" bạn

December 26, 2025 By Nguyễn An Khang

Chào bạn, chủ đề này cực kỳ "đời thường" và chạm đúng nỗi đau của bất kỳ team lập trình nào. Một bài viết tốt về chủ đề này không chỉ giúp người đọc cười (vì thấy mình trong đó) mà còn trang bị cho họ kỹ năng làm việc nhóm chuyên nghiệp.

Dưới đây là nội dung chi tiết cho bài blog, được viết theo phong cách hài hước nhưng đầy đủ kiến thức chuẩn (Conventional Commits).

Hãy tưởng tượng một kịch bản kinh dị lúc 3 giờ sáng: Server sập. Sếp gọi cháy máy. Bạn dò lại lịch sử code (git log) để xem ai đã sửa cái gì vào chiều nay mà gây ra nông nỗi này. Và đây là những gì bạn thấy:

commit a1b2c3d: fix bug
commit e4f5g6h: fix bug again
commit i7j8k9l: update
commit m1n2o3p: ...
commit q4r5s6t: final fix
commit u7v8w9x: final fix that (thề là lần cuối)

Lúc này, bạn có muốn tìm ngay địa chỉ nhà của chủ nhân những dòng commit kia để "tác động vật lý" không? Chắc chắn là có. Không ai biết "fix bug" là fix cái gì? Tại sao lại fix? Sửa ở file nào?

Viết Commit Message không chỉ là gõ cho xong để đẩy code lên. Đó là nghệ thuật giao tiếp. Hãy viết sao cho 6 tháng sau đọc lại, bạn không tự chửi chính mình.

Hôm nay, mình sẽ giới thiệu quy chuẩn Conventional Commits (Semantic Commits) - Tiêu chuẩn vàng để viết commit mà các công ty lớn đang dùng.

1. Tại sao phải viết chuẩn? (Ngoài việc để không bị đánh)

  • Dễ truy vết (Traceability): Nhìn vào lịch sử là biết ngay tính năng Login được thêm vào lúc nào, lỗi NullPointer được sửa ở đâu.
  • Review code nhanh hơn: Người review nhìn tiêu đề là biết mức độ ảnh hưởng của code.
  • Tự động hóa (Automation): Đây là phần "ăn tiền" nhất. Nếu bạn viết chuẩn, các công cụ CI/CD có thể tự động tạo ra Change Log (Danh sách thay đổi) xịn sò để gửi cho khách hàng mà bạn không cần gõ tay một chữ nào.

2. Công thức vàng: Cấu trúc của một Commit

Một commit chuẩn sẽ có dạng:

<type>(<scope>): <subject>
  • Type (Bắt buộc): Loại thay đổi (Code mới, sửa lỗi, hay dọn dẹp...).
  • Scope (Tùy chọn): Phạm vi thay đổi (Ví dụ: Auth, UserPage, Database).
  • Subject (Bắt buộc): Mô tả ngắn gọn (thường dưới 50 ký tự).

Ví dụ chuẩn: feat(auth): add google login button

3. Từ điển Type - Dân chuyên dùng từ gì?

Đừng dùng tiếng Việt không dấu hay Tiếng Anh bồi. Hãy dùng các từ khóa chuẩn sau (đặt ở đầu câu):

  • feat (Feature)
    Dùng khi bạn thêm một tính năng mới.
    Ví dụ: feat: add dark mode switch (Thêm nút chuyển chế độ tối).
  • fix (Bug fix)
    Dùng khi bạn sửa một lỗi.
    Ví dụ: fix: resolve crash when user uploads large image (Sửa lỗi crash khi upload ảnh lớn).
  • docs (Documentation)
    Dùng khi bạn chỉ thêm/sửa tài liệu (README, comment code) mà không đụng vào logic code.
    Ví dụ: docs: update API documentation for login endpoint.
  • refactor (Refactoring)
    Dùng khi bạn sửa code cho sạch hơn, đẹp hơn, tối ưu hơn nhưng không sửa lỗi và cũng không thêm tính năng mới.
    Ví dụ: refactor: simplify calculation logic in OrderService.
  • style
    Dùng khi bạn chỉ sửa khoảng trắng, dấu chấm phẩy, format lại code (Code linter).
    Ví dụ: style: format code with Prettier.
  • chore
    Dùng cho các việc lặt vặt không liên quan đến code sản phẩm (sửa file cấu hình build, cập nhật thư viện).
    Ví dụ: chore: update react to version 18.
  • perf (Performance)
    Dùng khi code của bạn giúp cải thiện hiệu năng.
    Ví dụ: perf: reduce image loading time by 50%.

4. Ba quy tắc vàng cho phần nội dung (Subject)

Quy tắc 1: Dùng thể mệnh lệnh (Imperative mood)

Hãy viết như bạn đang ra lệnh cho máy tính.

Sai: Added login button (Đã thêm...), Adding login button (Đang thêm...).

Đúng: Add login button (Thêm nút login).

Mẹo: Hãy nhẩm trong đầu câu này: "If applied, this commit will..." (Nếu được áp dụng, commit này sẽ...). -> If applied, this commit will Add login button. (Hợp lý!).

Quy tắc 2: Không viết hoa chữ cái đầu (Tùy chọn nhưng nên làm)

Để tiết kiệm chỗ và nhìn "geek" hơn, nhiều team quy định viết thường toàn bộ tiêu đề.

feat: add login thay vì feat: Add login.

Quy tắc 3: Không dùng dấu chấm câu ở cuối

Tiêu đề commit giống như tiêu đề bài báo, không cần dấu chấm.

fix: crash on start.

fix: crash on start

5. Level Nâng cao: Ép buộc cả Team phải tuân thủ

Bạn viết chuẩn, nhưng đồng nghiệp vẫn viết git commit -m "ok". Phải làm sao? Hãy dùng công cụ để chặn ngay từ cửa!

Nếu làm dự án JavaScript/Node.js/React, hãy cài 2 thư viện: Husky và Commitlint.

  • Husky: Một chú chó canh cổng. Mỗi khi ai đó gõ git commit, Husky sẽ chặn lại để kiểm tra.
  • Commitlint: Cảnh sát giao thông. Nó kiểm tra xem tin nhắn có đúng chuẩn type: subject không.

Nếu ông đồng nghiệp gõ: git commit -m "sửa lỗi tí". Husky sẽ báo lỗi đỏ lòm ngay tại Terminal và không cho phép commit. Thế là cả team bắt buộc phải viết chuẩn.

Kết luận

Viết Commit Message cũng giống như việc bạn đánh răng vậy. Ban đầu thấy phiền phức, nhưng làm quen rồi thì thấy nó thơm tho và sạch sẽ. Hãy bắt đầu từ hôm nay: feat:, fix:, refactor:. Đừng để những dòng commit vô nghĩa biến bạn thành kẻ tội đồ lúc 3 giờ sáng!
